## Challenge B: Docker Compose – Multicontainer-App

### Ziel
Eine einfache Multicontainer-Applikation mit Docker Compose deklarativ aufsetzen, testen und wieder entfernen.

### Was ist Docker Compose?
Docker Compose erlaubt es, mehrere Container als zusammenhängendes System zu definieren und zu starten. Die Konfiguration erfolgt deklarativ in einer `docker-compose.yaml`-Datei (IaC – Infrastructure as Code).

---

### 1. Anpassungen am compose.yaml

Port, Volumen und Netzwerk wurden gemäss Bedingungen angepasst:

```yaml
version: "3"
services:
  web:
    image: jannneururer/webapp_jann08_8091:1.0
    ports:
      - "5169:8080"   # Published Port: 5169, Target: 8080
    volumes:
      - neu-vol:/data
    networks:
      - neu-net

volumes:
  neu-vol:

networks:
  neu-net:
```

> **Hinweis:** `neu` entspricht den ersten 3 Buchstaben des Nachnamens (Bsp.: `neu-vol`, `neu-net`).

![pub port](/images/pub-port.png)

---

### Fachbegriffe

| Begriff           | Erklärung                                                                                      |
|-------------------|-----------------------------------------------------------------------------------------------|
| **Raft-Algorithmus** | Konsensalgorithmus der sicherstellt, dass alle Manager den gleichen Cluster-State kennen. Benötigt Mehrheit (Quorum) der Manager für Entscheidungen. |
| **Leader**        | Der aktive Manager-Node, der alle Entscheidungen trifft.                                       |
| **Reachable**     | Manager-Node ist erreichbar, aber nicht Leader. Übernimmt bei Ausfall des Leaders.            |
| **Drain**         | Node erhält keine neuen Container. Bestehende Container werden auf andere Nodes verschoben.   |
| **Active**        | Node ist bereit, Container zu empfangen und auszuführen.                                      |

## Challenge D: Docker Swarm – Imperativ

### Ziel
Einen Service imperativ auf dem Swarm-Cluster deployen, skalieren und Self-Healing testen.

---

### 1. Service erstellen (imperativ)

```bash

  --name 169-web \
  --replicas 5 \
  --publish 5010:80 \
  jannneururer/webapp_jann08_8091:1.0
```


### 4. Container manuell löschen (Self-Healing testen)

Auf Worker-Node 4 werden 3 Container gelöscht:

```bash
docker container rm <id1> <id2> <id3>
```

Danach auf einem Manager-Node prüfen:

```bash
docker service ps 169-web
```

**Beobachtung:** Docker Swarm startet die gelöschten Container automatisch neu, um den Desired State (10 Replicas) wiederherzustellen. Dieser Prozess nennt sich **Self-Healing** oder **Reconciliation Loop**.

---

### 7. Node-Ausfall simulieren (Aufgabe 3)

Worker-Node 4 (Node 4) wurde in der AWS Management Console **terminiert**.

Nach ca. 1 Minute:

```bash
docker stack services neu-zaehler
docker stack ps neu-zaehler
```

**Beobachtung:** Docker Swarm erkennt den Ausfall und startet die verlorenen Container automatisch auf dem verbleibenden Worker-Node 5 neu → **Self-Healing / Failover**.

![1terminated](/images/1terminated.png)

| Frage                                    | Antwort                                                  |
|------------------------------------------|----------------------------------------------------------|
| Desired State                            | 4 Replicas                                               |
| Observed State (nach Ausfall)            | Temporär weniger als 4, dann wieder 4 auf Node 5         |
| Was ist passiert?                        | Container von Node 4 wurden auf Node 5 neu gestartet     |
| Fachbegriff                              | **Self-Healing / Automatic Failover**                    |

---

```bash
docker stack rm neu-zaehler
```
