# KN05 - Container Monitoring & Security

## A) Logging von Containern

### Schritt 1: Container starten, Logs anzeigen, löschen

Container gestartet mit einfacher Ausgabe:
```bash
docker run --name m169_logtest ubuntu bash -c 'echo "Hallo"; echo "Fehler" >&2'
```

Logs angezeigt:
```bash
docker logs m169_logtest
```

Container entfernt:
```bash
docker rm m169_logtest
```

---

### Schritt 2: Laufende Logs anzeigen und zählen

Container im Hintergrund gestartet, der jede Sekunde "tick" ausgibt:
```bash
docker run -d --name m169_logtest ubuntu bash -c 'while true; do echo "tick"; sleep 1; done;'
```

Logs angezeigt (einmalig und fortlaufend):
```bash
docker logs m169_logtest
docker logs -f m169_logtest
```

Anzahl Log-Zeilen gezählt:
```bash
docker logs m169_logtest | wc -l
```

![alt text](</images/Screenshot 2026-05-05 082251.png>)

Container gestoppt und entfernt:
```bash
docker stop m169_logtest
docker rm m169_logtest
```

---

### Schritt 3: Syslog konfigurieren

Container gestartet, der Logs direkt ins syslog des Hosts schreibt:
```bash
docker run -d --log-driver=syslog ubuntu bash -c 'i=0; while true; do i=$((i+1)); echo "docker $i"; sleep 1; done;'
```

Syslog in Echtzeit überwacht:
```bash
sudo tail -f /var/log/syslog
```


## Fazit

Mit dem **deklarativen Ansatz** (Docker Compose) konnte die gesamte Monitoring-Umgebung mit einem einzigen Befehl aufgesetzt werden. Im Vergleich zur manuellen (imperativen) Installation spart dies viel Zeit und vermeidet Fehler. Die Lösung ist ausserdem wiederholbar und portierbar sie kann auf jeder Maschine gleich ausgeführt werden.

## C) Container Sicherheit
 
### Schritt 1: Image auf Sicherheitslücken scannen mit Trivy
 
Trivy installiert:
```bash
sudo apt install trivy -y
```
 
nginx-Image auf Schwachstellen gescannt:
```bash
trivy image nginx:latest
```
 
![alt text](</images/Screenshot 2026-05-05 093527.png>)
 
**Ergebnis:** Trivy zeigt alle gefundenen Schwachstellen mit Schweregrad (LOW, MEDIUM, HIGH, CRITICAL). Kritische Lücken sollten durch Aktualisierung des Images behoben werden.
 
---
 
### Schritt 2: Container ohne Root-Rechte starten
 
Container mit nicht-privilegierter Benutzer-ID gestartet:
```bash
docker run --user 1000:1000 nginx
```
 
Container mit read-only Filesystem (verhindert ungewollte Schreibzugriffe):
```bash
docker run --read-only --tmpfs /tmp:rw,size=64M nginx
```
 
![alt text](</images/Screenshot 2026-05-05 093651.png>)
 
---
 
### Schritt 3: AppArmor zur Laufzeitbeschränkung
 
AppArmor-Profil auf Container angewendet (begrenzt erlaubte Systemaufrufe):
```bash
docker run --security-opt "apparmor=docker-default" nginx
```
 
![alt text](</images/Screenshot 2026-05-05 093729.png>)

## Fazit
 
| Teil | Thema | Methode |
|------|-------|---------|
| A | Container Logging | `docker logs`, syslog |
| B | Monitoring | Docker Compose (deklarativ) |
| C | Sicherheit | Trivy, Minimalrechte, AppArmor |
 
**Wichtigste Erkenntnisse:**
- Docker schreibt Logs standardmässig als JSON. Mit `--log-driver=syslog` landen alle Logs zentral im System-Log.
- Docker Compose ermöglicht es, eine komplette Monitoring-Umgebung mit einem einzigen Befehl aufzusetzen.
- Container sollten nie als root laufen. Mit `--user`, `--read-only` und AppArmor kann man die Angriffsfläche stark reduzieren.
- Trivy hilft dabei, bekannte Sicherheitslücken in Images zu finden, bevor sie produktiv eingesetzt werden.
